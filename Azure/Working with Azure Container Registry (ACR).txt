# Technical Implementation (Step-by-Step)

### 1. Authentication & Environment Setup
First, I authenticated with Azure using the lab credentials and dynamically retrieved the assigned Resource Group name.

# Login to Azure
az login -u <username> -p <password>

# Get Resource Group name
export RESOURCE_GROUP=$(az group list --query "[0].name" -o tsv)



step 2 :  ACR Provisioning

# Created a private container registry with the Basic SKU in the eastus region.
az acr create --resource-group $RESOURCE_GROUP --name nautilusacr22771 --sku Basic --location eastus


step 3 : . Docker Image Lifecycle

The core of the step3  : 
Logging into the registry, building the image from a Dockerfile, and tagging it correctly for the ACR domain.

# Registry Authentication
az acr login --name nautilusacr22771

# Build and Tag the image
docker build -t nautilusacr22771.azurecr.io/nautilusacr22771:latest .

# Push to Azure Cloud
docker push nautilusacr22771.azurecr.io/nautilusacr22771:latest



step 4. Verification & Validation

Finally, I verified that the image was successfully stored in the registry repositories.

# List repositories and tags
az acr repository list --name nautilusacr22771 --output table
az acr repository show-tags --name nautilusacr22771 --repository nautilusacr22771 --output table


Challenges & Lessons Learned

    Tagging Precision: Ensuring the image tag matches the <registry-name>.azurecr.io format is critical for a successful push.
    Dynamic Variable Usage: Using --query to fetch the Resource Group name saves time and prevents manual errors in lab environments.
